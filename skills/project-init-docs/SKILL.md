# 项目文档初始化 Skill

## 核心原则

**必须先确认文档范围，再开始任何文档写作。** 不允许跳过文档范围确认阶段直接开始写作。

文档内容必须通过与用户的沟通讨论获得，不得凭空编造细节。每份文档采用迭代方式撰写和完善。

---

## 工作流程

### 第一阶段：了解项目

请用户用 2-3 句话描述项目，重点了解：
- 项目的核心功能是什么
- 预计规模（个人项目 / 小团队 / 多团队协作）
- 是否有数据库、外部 API 对接、复杂业务逻辑

### 第二阶段：评估并推荐文档清单

根据用户描述，按照以下规则评估项目规模并推荐文档，**必须附上每份文档的推荐理由**：

| 规模判断依据 | 默认推荐文档 |
|------------|-------------|
| 单人 / 功能单一 / 无复杂数据模型 | PRD + TDD |
| 多模块 / 有数据库 / 需要对接方 | + ERD + API |
| 多人协作 / 复杂业务 / 需要排期 | + SDD + 项目计划 |
| 微服务 / 多团队 / 合规需求 | + 00 项目概览 + 按需扩展 |

推荐格式示例：
```
根据您的描述，建议创建以下文档：

✅ 必要
- PRD（产品需求规格）—— 项目有明确的用户功能需求
- TDD（架构设计）—— 需要明确技术选型和系统结构

💡 建议
- ERD（数据库设计）—— 您提到有用户数据和订单数据

⏭️ 暂不需要
- SDD（详细设计）—— 当前规模较小，TDD 足够
- 项目计划 —— 单人项目可按需添加

是否按此清单进行？您可以要求添加或移除某些文档。
```

**等待用户明确确认或调整后，才能进入第三阶段。**

### 第三阶段：检查现有文档

确认文档清单后，先执行：
```bash
ls docs/design/ 2>/dev/null || echo "目录不存在"
```

如果目录已存在且有同名文档，询问用户是否覆盖。

创建目录（如不存在）：
```bash
mkdir -p docs/design
```

### 第四阶段：逐文档协作撰写

按确认的清单顺序，**每份文档**执行以下步骤：

1. **提问关键信息**：针对该文档类型，向用户提问必要信息（见各文档模板）
2. **输出初稿**：基于用户回答生成文档初稿
3. **等待反馈**：询问用户是否需要调整
4. **写入文件**：用户满意后写入对应路径
5. **更新注册表**：每写入一份文档，立即将其追加到 `docs/review-state.json` 的 `documents` 列表中

文档写入路径：`docs/design/<序号>_<类型>_<中文名>.md`

### 第五阶段：生成项目 README.md

所有设计文档撰写完成后，根据已生成的文档内容自动生成项目 `README.md`，包含：

- 项目名称与一句话简介（从 PRD 提取）
- 技术栈概览（从 TDD 提取）
- 文档索引表（链接到 `docs/design/` 下的所有文档）
- 快速开始（环境搭建、启动命令等，需向用户确认）

**README.md 也作为项目文档纳入注册表管理**，写入 `docs/review-state.json` 的 `documents` 列表，`type` 为 `README`。

### 第六阶段：写入文档注册表

所有文档写入完成后，确保 `docs/review-state.json` 的 `documents` 字段完整记录了本次创建的所有文档。

注册表写入规则：
- 如果 `docs/review-state.json` 不存在，创建文件并写入完整结构（见下方格式）
- 如果文件已存在，只更新 `documents` 字段，**不覆盖** `last_reviewed_commit` 等 review 相关字段

**注册表文档条目格式**：
```json
{
  "path": "docs/design/01_PRD_产品需求规格说明书.md",
  "type": "PRD",
  "description": "产品需求规格，包含用户故事和验收标准",
  "created_at": "YYYY-MM-DD"
}
```

**完整文件结构**（首次创建时）：
```json
{
  "last_reviewed_commit": null,
  "last_reviewed_commit_short": null,
  "last_review_date": null,
  "last_review_summary": null,
  "documents_updated": [],
  "documents": [
    {
      "path": "docs/design/01_PRD_产品需求规格说明书.md",
      "type": "PRD",
      "description": "产品需求规格，包含用户故事和验收标准",
      "created_at": "YYYY-MM-DD"
    }
  ]
}
```

## 文档模板与关键提问

详细模板请参考：`references/doc-templates.md`

各文档写作前必须向用户确认的核心问题：

**PRD**：目标用户是谁？核心功能有哪几个？有哪些明确不做的边界？

**TDD**：技术栈是什么？系统有哪些主要模块？部署方式（本地 / 云服务 / Docker）？

**ERD**：有哪些核心实体（如用户、订单、商品）？它们之间的关系？

**API**：是 RESTful 还是 GraphQL？有哪些核心接口？需要鉴权吗？

**SDD**：哪些模块逻辑最复杂？有没有特殊算法或状态机需要说明？

**项目计划**：预计上线时间？几个主要里程碑？当前团队规模？

---

## 文档命名规范

- 序号从 `01` 开始，两位数补零
- 类型用英文缩写（PRD / TDD / ERD / API / SDD）
- 中文名描述文档内容
- 示例：`01_PRD_产品需求规格说明书.md`、`04_API_接口文档.md`

如有项目概览文档，命名为 `00_项目概览.md`。

---

## 注意事项

- 所有文档使用**中文**撰写
- 文档内容来自用户确认，不编造业务细节
- 每份文档完成后询问用户"是否继续下一份文档"
- 如用户中途想修改已完成的文档，随时响应，完成后继续未完成的文档
- 全部文档完成后，输出文档索引表供用户参考（README.md 中已包含，此处仅作确认）
- README.md 是项目文档的一部分，纳入 `docs/review-state.json` 注册表管理，后续 code-review-doc-sync skill 会同步追踪其更新
- `docs/review-state.json` 是与 code-review-doc-sync skill 共享的状态文件，每次新增文档后必须同步更新，不得遗漏

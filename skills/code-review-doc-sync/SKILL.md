# 代码审查 + 文档同步 Skill

## 核心原则

- **只输出 review 结论，不修改任何代码文件**
- 文档更新必须经用户确认后才能写入
- 跨会话通过状态文件记忆上次 review 位置

---

## 工作流程

### 第一步：确定 review 范围

**读取状态文件**（记录上次 review 位置）：
```bash
cat docs/review-state.json 2>/dev/null || echo "首次 review"
```

**获取最近提交列表**：
```bash
git log --oneline -20
```

根据以下规则确定范围：

| 情况 | 默认行为 |
|------|---------|
| 有状态文件 | 从 `last_reviewed_commit` 到 HEAD |
| 无状态文件（首次） | 询问用户，默认最近 1 次提交 |
| 用户指定范围 | 按用户指定执行，如"最近 3 次"、"从 abc123 到现在" |

向用户确认 review 范围后再继续。

**获取变更内容**：
```bash
# 指定 commit 范围
git diff <start_commit> <end_commit>

# 从上次 review 到现在
git diff <last_reviewed_commit> HEAD

# 最近 N 次提交
git diff HEAD~N HEAD

# 同时获取变更文件列表
git diff --name-status <start> <end>
```

---

### 第二步：代码 Review

对变更内容进行标准审查，**只输出结论，不修改代码**。

审查维度：

**代码质量**
- 逻辑正确性：是否有明显的逻辑错误或边界条件遗漏
- 代码可读性：命名是否清晰，函数职责是否单一
- 重复代码：是否有可以抽取复用的逻辑
- 错误处理：异常 / 错误是否被妥善处理

**代码规范**
- 命名风格是否与项目一致（驼峰 / 下划线等）
- 注释是否充分，特别是复杂逻辑
- 函数/类的长度是否合理
- 是否有调试代码、临时注释未清理

**潜在风险**
- 安全问题：SQL 注入、XSS、敏感信息硬编码等
- 性能问题：N+1 查询、不必要的循环、大数据量处理
- 并发问题：共享状态、竞态条件

**Review 报告输出到文件**：`docs/code-review.md`

报告格式设计为**面向 AI 阅读**，每个问题必须给出明确的修改位置和修改方式，使 AI 可以直接据此修改代码。输出前先询问用户确认保存路径，默认为 `docs/code-review.md`。

详细格式规范见 `references/code-review-template.md`，核心结构为：

1. **问题总览表** — 按优先级（🔴 必须修复 → 🟡 建议改进 → 🟢 可选优化）列出所有问题的表格，包含编号、优先级、文件、行号、问题概述
2. **问题详情** — 按总览表编号逐一展开，每个问题包含：当前代码、修改后代码、修改说明

**输出规则**：
- 只列出问题，不包含亮点、总体评价等非问题内容
- 问题总览表按优先级排列，同一优先级内按严重程度从高到低
- 每个问题必须包含**当前代码**和**修改为**两个代码块，供 AI 直接对比执行修改
- 位置信息必须精确到行号（如 `L42` 或 `L88-95`）
- 每次写入直接覆盖文件旧内容，不追加、不保留历史问题
- 输出后询问用户："Review 报告已写入 docs/code-review.md，是否继续进行文档同步分析？"

---

### 第三步：加载文档注册表

用户确认继续后，从状态文件中读取文档注册表（已在第一步读取，直接使用 `documents` 字段）。

**根据读取结果分三种情况处理：**

#### 情况 A：注册表存在且有文档记录
直接使用注册表中的文档列表，**不重新扫描**。向用户展示当前注册的文档：

```
当前项目文档（来自注册表）：
- docs/design/01_PRD_产品需求规格说明书.md — PRD：产品需求规格
- docs/design/03_ERD_数据库设计文档.md — ERD：数据库设计
- README.md — 项目说明
```

同时验证注册表中的文件是否实际存在，检测失效记录：
```bash
# 逐一检查注册表中每个 path 是否存在
ls <path> 2>/dev/null || echo "NOT_FOUND: <path>"
```

如发现文件不存在（可能被删除或重命名），提示用户：
```
⚠️ 以下文档在注册表中登记，但文件已不存在：
- docs/design/02_TDD_架构设计文档.md

可能已被删除或重命名，是否从注册表中移除？
```

等用户确认后，从 `documents` 数组中删除对应条目并更新状态文件。

#### 情况 B：注册表不存在，或 `documents` 字段为空
提示用户，并询问如何处理：

```
未找到文档注册表（或注册表中没有文档记录）。

有两个选项：
1. 立即扫描项目中所有 .md 文件，建立注册表
2. 手动指定需要纳入管理的文档路径

请问您希望如何处理？
```

**选项 1 — 全量扫描并建立注册表**：
```bash
find . -name "*.md" \
  -not -path "*/node_modules/*" \
  -not -path "*/.git/*" \
  -not -path "*/vendor/*" \
  -not -path "*/dist/*" \
  -not -path "*/build/*" \
  | sort
```

扫描完成后，列出所有找到的文档，让用户确认哪些纳入注册表管理，并补充每份文档的 `type` 和 `description`，写入 `docs/review-state.json`。

**选项 2 — 手动指定**：
用户提供路径列表，逐一确认 `type` 和 `description`，写入注册表。

#### 情况 C：用户主动要求重新扫描 / 补充文档
触发词：「重新扫描文档」、「更新文档列表」、「添加文档 <路径>」、「注册文档」等。

- **重新扫描**：执行情况 B 的选项 1 流程，将扫描结果与现有注册表合并（已有记录保留，新发现的询问是否添加）
- **补充指定文档**：直接将用户提供的路径追加到 `documents` 数组，补充 `type` 和 `description` 后写入状态文件

---

### 第四步：分析文档影响

对照变更文件列表，结合注册表中每份文档的 `type` 字段，按以下规则判断哪些文档需要更新：

| 代码变更类型 | 判断依据 | 可能影响的文档 |
|------------|---------|-------------|
| 新增 / 修改 API 路由、控制器 | 文件路径含 `route`、`controller`、`handler`、`api` | API 接口文档 |
| 修改数据库 Model / Migration / Schema | 文件路径含 `model`、`migration`、`schema`、`entity` | ERD 数据库设计 |
| 新增模块 / 重构目录结构 | 新增目录、大量文件移动 | TDD 架构设计、SDD 详细设计 |
| 修改核心业务逻辑 / 服务层 | 文件路径含 `service`、`logic`、`domain` | SDD 详细设计 |
| 修改配置 / 部署相关文件 | `Dockerfile`、`docker-compose`、`nginx.conf`、`.env.example` | TDD 架构设计 |
| 完成功能里程碑 | 提交信息含 `feat`、`feature`、`完成`、`上线` | 项目计划 |
| 修改认证 / 权限逻辑 | 文件路径含 `auth`、`permission`、`middleware` | API 接口文档（认证说明）|

**文档影响分析输出格式**：

```
## 文档影响分析

根据本次变更，以下文档需要更新：

### 需要更新的文档

**📄 docs/design/04_API_接口文档.md**
- 原因：新增了 /users/profile 接口
- 建议更新内容：添加该接口的请求/响应说明

**📄 docs/design/03_ERD_数据库设计文档.md**
- 原因：users 表新增了 avatar_url 字段
- 建议更新内容：在 users 表字段说明中添加 avatar_url

### 无需更新的文档
- docs/design/01_PRD_产品需求规格说明书.md — 功能需求无变化
- README.md — 无相关变更
```

如果某个变更涉及但对应文档**不存在**，需明确提示：
```
⚠️ 发现 API 变更，但项目中没有找到 API 接口文档。
建议使用 project-init-docs skill 创建 docs/design/04_API_接口文档.md。
```

---

### 第五步：逐文档确认并更新

对每份需要更新的文档：

1. 展示具体的更新建议（diff 形式或说明新增/修改内容）
2. 询问用户是否按此更新
3. 用户确认后写入文档
4. 在文档末尾的**变更记录**表中追加一条记录：

```markdown
| v1.x | YYYY-MM-DD | 根据 commit <hash> 更新：xxx |
```

---

### 第六步：更新状态文件

所有操作完成后，更新 `docs/review-state.json`，**必须完整保留 `documents` 字段**，只更新 review 相关字段：

```json
{
  "last_reviewed_commit": "<结束 commit 的完整 hash>",
  "last_reviewed_commit_short": "<短 hash>",
  "last_review_date": "YYYY-MM-DD",
  "last_review_summary": "简要描述本次 review 涉及的主要变更",
  "documents_updated": ["docs/design/04_API_接口文档.md"],
  "documents": [
    {
      "path": "docs/design/01_PRD_产品需求规格说明书.md",
      "type": "PRD",
      "description": "产品需求规格，包含用户故事和验收标准",
      "created_at": "YYYY-MM-DD"
    },
    {
      "path": "docs/design/04_API_接口文档.md",
      "type": "API",
      "description": "REST API 接口文档，包含认证说明和所有端点定义",
      "created_at": "YYYY-MM-DD"
    }
  ]
}
```

获取完整 commit hash：
```bash
git rev-parse HEAD
```

**写入前必须先读取现有文件内容，在原有基础上合并更新，严禁直接覆盖导致 `documents` 数据丢失。**

## --fix 模式：读取报告并修复问题

当用户使用 `--fix` 参数时，跳过 review 流程，直接读取已有的 code-review 报告文件，逐项修复问题。

触发方式：`--fix`、`--fix <文件路径>`、用户说「修复 review 问题」、「按报告修复」

### fix 第一步：读取报告

读取 code-review 报告文件：

```bash
cat docs/code-review.md
```

如果用户指定了文件路径，则读取指定文件。

如果文件不存在，提示用户：

```
未找到 code-review 报告文件。请先执行 review 生成报告，或指定报告文件路径。
```

### fix 第二步：解析问题总览

解析报告中的**问题总览表**（文件中只有待处理问题），向用户展示：

```
从 code-review.md 中读取到以下问题：

| # | 优先级 | 文件 | 问题概述 |
|---|--------|------|----------|
| 1 | 🔴 必须修复 | path/to/file.js | ... |
| 2 | 🟡 建议改进 | path/to/file.js | ... |

按优先级从高到低逐项修复。是否开始？
```

### fix 第三步：逐项修复

按优先级从高到低（🔴 → 🟡 → 🟢），逐项执行：

1. **读取问题详情**：从报告中提取该问题的当前代码、修改目标、修改说明
2. **读取目标文件**：确认文件存在，定位到对应行号
3. **验证代码**：对比报告中记录的「当前代码」与文件中实际代码是否一致。如果不一致，提示用户代码可能已变更，展示差异后确认是否继续
4. **执行修改**：按报告中的「修改为」内容修改代码
5. **立即删除已处理问题**：从 `docs/code-review.md` 的问题总览表和问题详情中**删除**该条目，剩余问题重新编号并写回文件
6. **向用户报告**：简要说明已修复的内容，继续下一个问题

每修复一个问题后，可选择：
- **逐项确认模式**（默认）：每修复一项后暂停，等用户确认再继续下一项
- **批量修复模式**：用户说「全部修复」时，连续修复所有待处理问题，最后统一汇报

对于用户选择跳过/忽略的问题，同样立即从文件中删除。

### fix 第四步：汇总

每处理一个问题就即时更新文件，最后输出汇总：

```
本次处理：
- ✅ 已修复：N 项
- ⏭️ 跳过：N 项
- ⏳ 剩余待处理：N 项

docs/code-review.md 已更新。
```

如果文件中没有剩余问题，保留空文件即可。

---

## 注意事项

- **只输出 review 结论，不修改任何代码文件**（`--fix` 模式除外）
- 文档更新采用**追加/修改**方式，不得删除原有内容（除非用户明确要求）
- 每次写入状态文件前必须先读取现有内容，合并更新，**严禁覆盖 `documents` 字段**
- 如果项目没有 `docs/` 目录，在写入状态文件前先创建：`mkdir -p docs`
- review 报告默认写入 `docs/code-review.md`，面向 AI 阅读格式，包含精确的修改位置和修改代码
- `docs/code-review.md` 纳入文档注册表管理，`type` 为 `CODE_REVIEW`
- 遇到无法判断是否影响文档的变更，保守地列出可能受影响的文档，由用户决定
- 用户说「重新扫描」、「更新文档列表」、「添加文档」时，直接跳到第三步情况 C 处理，无需完整执行 review 流程
